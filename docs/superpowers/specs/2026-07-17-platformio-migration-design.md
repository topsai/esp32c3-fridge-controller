# ESP32-C3 冰箱控制器 PlatformIO 迁移设计

## 1. 项目目标

在 `C:\Users\John\Desktop\Code\Fridge` 中建立一个全新的 PlatformIO Arduino 工程，以
`C:\Users\John\Desktop\Arduino_Projects\Fridge1\Fridge1.ino` 的当前工作区版本为唯一功能基准，
面向 ESP32-C3 完成工程迁移、构建验证、Git 初始化、完整文档编写和 GitHub 公开发布。

新 GitHub 仓库名称为 `esp32c3-fridge-controller`。新工程不继承原 Arduino 工程的 Git 历史。

## 2. 核心约束

- 保持现有运行逻辑，不主动修改温控策略、回差、定时、故障处理或人机交互行为。
- 保持现有引脚定义、启动状态、继电器与风扇电平语义、NVS 键名和 OLED 地址。
- 不进行模块拆分、类封装、命名清理或无关重构。
- 只允许进行 PlatformIO 编译所必需的机械性调整，并逐项记录。
- 原 `.ino` 当前工作区版本优先于其 Git 历史版本。
- PlatformIO 编译通过不等同于硬件验证；实机验证结果不得凭空宣称。

## 3. 技术方案

采用保守迁移方案：将原 `.ino` 内容迁入 `src/main.cpp`。Arduino 构建系统会自动为 `.ino`
生成函数原型，而普通 C++ 源文件不会，因此只有在编译器确实要求时，才在 `main.cpp` 中补充函数声明。
除此之外不改变控制流程。

PlatformIO 环境使用：

- Platform：`espressif32`
- Framework：`arduino`
- Board：`esp32-c3-devkitm-1`
- 依赖：OneWire、DallasTemperature、OneButton、Adafruit GFX Library、Adafruit SSD1306

依赖版本将使用与当前 ESP32 Arduino 平台兼容的明确约束，避免因未来上游更新导致不可重复构建。

## 5. 固件行为保护

迁移后应保留以下可观察行为：

- DS18B20 使用 GPIO 2，NTC 使用 GPIO 4。
- 继电器使用 GPIO 3，风扇使用 GPIO 10。
- 两个按键使用 GPIO 1 和 GPIO 7。
- OLED I2C 使用 GPIO 5/6，地址为 `0x3C`，分辨率为 128×32。
- 温度设定值以 0.5 °C 为步进，范围保持 -30 °C 至 20 °C，默认 -10 °C。
- 设定值继续存储在 NVS 命名空间 `fridge` 的 `setpoint_x2` 键中。
- NTC 采样、EWMA 滤波、DS18B20 非阻塞读取、故障停机、OLED 超时和日志节奏保持不变。
- 上电后的继电器初始行为及当前压缩机/风扇控制语义保持不变。

文档会如实描述代码的实际行为。如果注释与代码行为冲突，以代码为准，并在迁移说明中指出，
但不借此修改逻辑。

## 7. 验证策略

最低验证集合为：

1. 对迁移前后的有效固件内容进行差异审查，确认只有文件形态和编译必需调整。
2. 执行 `pio run`，要求目标环境完整编译并返回退出码 0。
3. 检查 PlatformIO 依赖解析结果及最终固件尺寸输出。
4. 检查 Git 工作区、已跟踪文件、提交内容和远程地址。
5. 推送后检查 GitHub 仓库可访问性、默认分支和提交一致性。

实机项目应另外执行冷启动、两路温度传感器、按键短按/长按、OLED 熄屏唤醒、NVS 断电保持、
继电器/风扇输出、传感器断线保护等测试。若本次没有连接硬件，这些项目只作为待执行清单记录。
