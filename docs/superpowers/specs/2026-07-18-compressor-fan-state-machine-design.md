# 压缩机与风扇状态机设计

## 目标

在不改变温度阈值、传感器、按键、OLED 或 NVS 行为的前提下，将压缩机与风扇输出改为显式状态机。
压缩机停止后风扇继续运行 5 分钟，为压缩机散热，然后自动关闭。

## 状态模型

压缩机状态：

- `Off`：继电器输出低电平。
- `Running`：继电器输出高电平。

风扇状态：

- `Off`：风扇输出低电平。
- `FollowingCompressor`：压缩机运行，风扇输出高电平。
- `Cooldown`：压缩机已经停止，风扇继续输出高电平，等待 300000 ms 到期。

状态机保存压缩机状态、风扇状态和最近一次有效停机转换的时间戳。

## 事件与转换

### 请求压缩机运行

- `Off → Running`：压缩机输出高，风扇进入 `FollowingCompressor` 并输出高。
- `Running → Running`：保持状态，不产生额外转换。
- `Cooldown → FollowingCompressor`：取消当前冷却等待，风扇持续运行。

### 请求压缩机停止

- `Running → Off`：压缩机输出低，记录当前时间，风扇进入 `Cooldown` 并保持高电平。
- `Off → Off`：保持现有风扇状态；重复停止请求不得重置冷却起始时间。

温控达到下限和 NTC 故障停机均使用同一停止事件，因此两者都执行 5 分钟散热。

### 周期计时

- `Cooldown` 且经过时间小于 300000 ms：保持风扇高电平。
- `Cooldown` 且经过时间大于或等于 300000 ms：风扇进入 `Off` 并输出低电平。
- 其他状态：不发生计时转换。

经过时间使用 `uint32_t(now - cooldownStartedAt)` 计算，使 `millis()` 在约 49.7 天回绕时仍保持正确语义。

## 代码边界

- `include/fridge_state_machine.h` 保存不依赖 Arduino API 的状态、事件和纯转换逻辑。
- `src/main.cpp` 将 `millis()` 和目标运行请求传入状态机，并根据状态集中写入 GPIO。
- OLED 通过状态机查询压缩机与风扇是否运行，不再依赖可与状态脱节的独立布尔值。
- 温控调用点继续使用 `setRelayPhysical(bool)`，函数签名保持不变，以限制改动范围；该函数内部改为向状态机发送请求。
- `updateFanLogic()` 只推进计时并应用输出，不再读取继电器 GPIO 来推断状态。

## 启动行为

状态机对象初始为压缩机 `Off`、风扇 `Off`。`setup()` 仍调用 `setRelayPhysical(true)`，因此上电后的既有行为仍为压缩机和风扇开启，不产生虚假的冷却倒计时。

## 验证

编译期状态转换契约覆盖：

- 初始状态；
- 开机时压缩机与风扇同步运行；
- 正常停机进入冷却；
- 299999 ms 时保持冷却；
- 300000 ms 时风扇关闭；
- 重复停机请求不重置时间；
- 冷却期间重新开机；
- `millis()` 回绕边界。

另外执行现有 OLED 回归测试、完整 PlatformIO 构建、文档一致性检查，并在实机验证清单中加入风扇延时项目。

## 非目标

- 不增加压缩机最小启停时间保护。
- 不改变继电器和风扇的有效电平。
- 不改变故障判定、温控回差或采样周期。
- 不增加可配置的冷却时长或新的菜单。
