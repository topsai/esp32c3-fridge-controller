# 单芯片串口 HIL 实时测试系统设计

## 1. 目标

通过 USB 串口直接向冰箱控制器 ESP32-C3 发送指令，在连接真实控制板的情况下实时注入传感器值、
触发用户操作、读取内部状态、执行自动测试场景并生成报告。不使用第二块控制芯片。

测试系统验证固件业务逻辑、状态转换和 ESP32 输出电平状态，但明确区分无法自动确认的真实传感器接线、
OLED 像素、继电器触点、风扇转动和压缩机动作。

## 2. 构建隔离

保留现有正式环境 `esp32-c3-devkitm-1`，新增 `esp32-c3-hil` 环境并定义 `FRIDGE_HIL`。

- 正式环境不编译串口命令解析、传感器注入或测试事件接口。
- HIL 环境始终以 115200 baud 初始化串口。
- HIL 环境上电时默认锁定危险输出，压缩机和风扇保持关闭。
- 只有收到显式解锁指令后，状态机输出才允许写入真实 GPIO。
- 断开串口、命令超时或执行复位时重新进入锁定状态。

该隔离防止生产固件通过串口被意外控制，也防止测试程序启动后立即带动真实负载。

## 3. 固件结构

### 3.1 输入来源

NTC 和 DS18B20 分别具有两种输入模式：

- `Physical`：读取真实硬件。
- `Injected`：使用串口指定的有效温度或故障值。

注入发生在传感器读取结果进入现有温控与显示逻辑之前。温控算法本身不区分数据来源，确保测试覆盖真实生产路径。

### 3.2 用户事件

串口可以触发升温/降温短按和长按。事件调用与 OneButton 回调相同的业务函数，但不宣称验证了实体按键、
GPIO 输入或 OneButton 去抖逻辑。

长按指令使用持续时间参数，由 HIL 任务按现有约 200 ms 节奏产生重复事件，测试结果应与实体长按业务行为一致。

### 3.3 状态快照

状态快照至少包含：

- 固件版本和 HIL 协议版本；
- 设备运行时间；
- NTC/DS18B20 输入模式、原始测试值和业务层最终值；
- 设定温度；
- 压缩机和风扇状态；
- 风扇冷却剩余毫秒数；
- NTC 故障状态与错误计数；
- OLED 开关状态；
- 输出锁定状态；
- GPIO 3 和 GPIO 10 的期望输出电平；
- 最近命令序号和结果。

## 4. 串口协议

协议采用逐行 ASCII 请求和逐行 JSON 响应。ASCII 请求便于人工调试，JSON 响应便于 Python 严格解析。

请求格式：

```text
HIL <sequence> <command> [arguments...]
```

成功响应：

```json
{"type":"ack","sequence":12,"ok":true}
```

失败响应：

```json
{"type":"ack","sequence":12,"ok":false,"error":"invalid_argument"}
```

状态和事件响应同样携带序号。每个请求必须恰好产生一个确认响应；`STATUS` 的确认响应内嵌完整状态，避免多行关联歧义。

### 4.1 指令集合

```text
HIL <seq> PING
HIL <seq> STATUS
HIL <seq> OUTPUTS UNLOCK
HIL <seq> OUTPUTS LOCK
HIL <seq> NTC PHYSICAL
HIL <seq> NTC VALUE <celsius>
HIL <seq> NTC FAULT
HIL <seq> DS PHYSICAL
HIL <seq> DS VALUE <celsius>
HIL <seq> DS FAULT
HIL <seq> BUTTON UP CLICK
HIL <seq> BUTTON DOWN CLICK
HIL <seq> BUTTON UP LONG <milliseconds>
HIL <seq> BUTTON DOWN LONG <milliseconds>
HIL <seq> DISPLAY WAKE
HIL <seq> NVS CLEAR
HIL <seq> REBOOT
HIL <seq> RESET
```

`NVS CLEAR`、`REBOOT` 和 `OUTPUTS UNLOCK` 属于受保护命令，Python 端必须显示明确提示，自动场景必须显式声明允许使用。

### 4.2 校验与限制

- 最大命令行长度固定，超长输入返回错误并清空当前行。
- 温度参数只接受有限浮点数及项目支持范围内的测试扩展值。
- 长按持续时间设置上限，防止串口错误造成无限事件。
- 未知命令、参数缺失、多余参数和非法序号均返回稳定错误码。
- 解析器不得使用动态 JSON 库；响应使用固定缓冲区输出，降低堆碎片风险。

## 5. 安全联锁

HIL 固件默认锁定输出。锁定状态下：

- GPIO 3 和 GPIO 10 强制为关闭电平；
- 内部状态机仍可运行并在快照中报告“期望输出”，用于无负载逻辑测试；
- 只有 `OUTPUTS UNLOCK` 允许真实 GPIO 跟随状态机；
- Python UI 使用持续醒目的红色标识显示已解锁状态；
- 程序退出时尝试发送 `OUTPUTS LOCK`；固件端还使用命令看门狗，在失去心跳后自动锁定。

硬件连接压缩机或市电时，不允许自动执行输出解锁场景。真实负载测试必须作为人工确认步骤。

## 6. Python 测试工具

Python 工具拆为四个边界清晰的组件：

- `transport.py`：串口发现、连接、序号、超时、重试和逐行接收。
- `protocol.py`：构造请求、解析 JSON、验证响应字段和错误码。
- `runner.py`：执行场景步骤、断言、计时、清理和报告收集。
- `dashboard.py`：实时状态卡片、曲线、手动指令和场景启动界面。

首版仪表板使用 Python 标准库 `tkinter`，曲线使用 Matplotlib，串口使用 pyserial。测试核心不依赖 UI，命令行和 CI 可以运行同一场景。

## 7. 自动场景

场景以版本化 JSON 文件描述，包含前置条件、步骤、超时、断言、是否允许危险命令和清理动作。

首批场景覆盖：

1. 协议握手、非法命令和状态结构。
2. 输出默认锁定、解锁、重新锁定和心跳超时。
3. NTC 高低阈值、回差、故障累计和恢复。
4. DS18B20 有效值、故障值及显示状态数据。
5. 升降温短按、长按、上下限和熄屏唤醒。
6. NVS 保存、重启恢复和受保护清除。
7. 压缩机运行、正常停机、风扇 5 分钟冷却、到期关闭和中途重启。
8. OLED 60 秒超时状态和唤醒状态。

默认场景不等待真实 5 分钟之外的超长周期，也不引入虚拟系统时钟；5 分钟冷却场景按真实时间运行，确保测试实际计时路径。

## 8. 实时仪表板

界面显示：

- 连接端口、固件/协议版本和最后响应时间；
- NTC、DS18B20 和设定温度实时曲线；
- 压缩机、风扇、冷却剩余时间、故障和 OLED 状态；
- 输出锁定状态；
- 当前场景、步骤、耗时、通过/失败数量；
- 原始协议日志。

界面允许选择串口、连接/断开、发送安全手动指令、运行单个场景或完整测试集，并导出报告。

## 9. 报告

每次运行生成：

- JSON：机器可读的环境、步骤、命令、响应、断言和时间戳；
- HTML：面向人工查看的摘要、失败详情和状态曲线图片。

报告目录默认加入 `.gitignore`，场景定义和报告模板纳入 Git。

## 10. 测试策略

- 固件协议解析器使用编译期/主机构建可覆盖的纯函数测试。
- Python 协议、场景和报告使用 `unittest`，串口层使用内存传输替身。
- 保留现有状态机编译期契约和 OLED 回归测试。
- 两个 PlatformIO 环境都必须完整构建成功。
- 连接硬件后运行握手、安全锁、输入注入、状态观测和完整场景。
- 实际 OLED 像素、实体按键、传感器电气接口及负载动作列为人工检查，不能由注入测试冒充通过。

## 12. 完成标准

- 正式环境不暴露 HIL 指令且构建成功。
- HIL 环境支持完整指令集、默认输出锁和结构化状态响应。
- Python 命令行场景运行器、实时仪表板和 JSON/HTML 报告可用。
- 自动化测试覆盖协议、错误处理、核心注入场景和报告生成。
- 连接 ESP32-C3 后可以实时查看状态并运行安全的自动测试集。
- 文档明确接线、刷写、操作、安全边界和人工验证项目。
